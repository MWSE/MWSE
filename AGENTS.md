## C++ Includes

In the MWSE and CSSE projects, Forced Include Files are used for standard libraries and external dependencies through `stdafx.h` / `pch.h`.

Do not add standard-library or external-dependency `#include`s directly to `.h` or `.cpp` files in these projects.


## File/Module Structure

This repo is mostly flat inside the main C++ project directories. Use filename prefixes as the primary map.

```
MWSE/                    # Main MWSE DLL (C++)
  main.cpp               # DLL entry point, startup checks, core hooks, MGE exports
  LuaManager.*           # Lua VM setup, library registration, event hook registration
  PatchUtil.*            # Central patch install/uninstall and hook helpers
  TES3*.h/.cpp           # Morrowind runtime object wrappers and methods
  TES3*Lua.h/.cpp        # Lua bindings for TES3 wrappers
  Lua*Event.h/.cpp       # Lua event payload/trigger types
  Lua*.h/.cpp            # Other Lua runtime helpers, objects, timers, executor, utilities
  MGE*, Mge*             # MGE API bindings and MGE/MWScript VM integration
  Instruction*.h/.cpp    # MWScript instruction registration/interface glue
  mw*.h/.cpp             # MWScript adapter, offsets, and legacy support helpers
  NI*.h/.cpp             # NetImmerse/Gamebryo wrappers used by MWSE Lua/runtime code
  CrashLog*.hpp/.cpp     # Crash logging subsystem
  *Util.h/.cpp           # Local utility modules such as FileUtil, MemoryUtil, LuaUtil
  MWSE.rc                # Windows resources for the MWSE DLL

CSSE/                    # Construction Set Extender (C++)
  CS*.h/.cpp             # Construction Set object wrappers, usually mirroring TES3 concepts
  Dialog*.h/.cpp         # Editor dialog patches and dialog-specific behavior
  WindowMain.*           # Main editor window patches/commands
  ObjectWindowTabData.*  # Object-window tab/column behavior
  RenderWindow*.h/.cpp   # Render-window scene, selection, and widget behavior
  *Util.h/.cpp           # CSSE-local utilities
  CSSE.rc, res/          # Construction Set resources

SharedSE/                # Shared code compiled into both MWSE and CSSE
  NI*.h/.cpp             # Shared NetImmerse/Gamebryo primitives and containers
  *Util.h/.cpp           # Shared low-level utilities such as MemoryUtil and CodePatchUtil
  StlList/NITArray/etc.  # Shared container wrappers and helpers

MWSE-Tests/              # C++ unit tests (Visual Studio C++ Unit Test Framework)
MWSE-Update/             # C# auto-updater
deps/                    # Vendored C/C++ dependencies
packages/                # NuGet package restore output
misc/package/            # Release package staging files

docs/
  source/
    apis/                # Generated Lua API reference pages
    events/              # Generated Lua event reference pages
    types/               # Generated Lua type reference pages
    references/          # Miscellaneous reference pages
    guides/              # How-to guides
    assets/              # Documentation assets
  *-definitions-guide.md # Guides for writing API/event/type/operator definitions

autocomplete/
  definitions/
    global/              # Global Lua library/API definitions
    namedTypes/          # Named type definitions for MWSE, MGE, NI, TES3, TES3UI, etc.
    events/standard/     # Lua event autocomplete definitions and examples
  builders/              # Scripts that generate autocomplete output
  lua/                   # Lua runtime used by autocomplete builders
```

**Generated Lua docs:**
- `docs/source/apis/`, `docs/source/events/`, and `docs/source/types/` are generated from `autocomplete/definitions/`.
- Edit the autocomplete definition source first: `global/` for APIs, `events/standard/` for events, and `namedTypes/` for types and type members.
- Regenerate headless from `autocomplete/`:
  ```powershell
  $env:LUA_PATH = ".\?.lua;.\?\init.lua;.\lua\?.lua;.\lua\?\init.lua"
  $env:LUA_CPATH = ".\?.dll;.\lua\?.dll"
  .\lua\lua.exe .\builders\emmy.lua
  .\lua\lua.exe .\builders\mkdocs.lua
  ```
- The builders delete and recreate generated output directories. If sandboxing blocks that, request permission and rerun the same commands.
- Generated markdown files include an autogenerated warning. Do not hand-edit those files.

**Headless build:**
- Run this from the repo root:
  ```powershell
  $msbuild = & "${env:ProgramFiles(x86)}\Microsoft Visual Studio\Installer\vswhere.exe" -latest -products * -requires Microsoft.Component.MSBuild -find "MSBuild\**\Bin\MSBuild.exe" | Select-Object -First 1
  & $msbuild MWSE.sln /t:Build /p:Configuration=Debug /p:Platform=x86 /nr:false
  ```
- Use `/p:Configuration=Release` instead of `Debug` for a release build.
- Build outputs go under `build/Debug/` or `build/Release/`. Post-build steps also copy DLL/PDB files to `MorrowindDir` from `SharedSE/MWSE.User.props`; that directory must exist.

**Naming conventions:**
- TES3 game objects: `TES3Foo.h` / `TES3Foo.cpp` + `TES3FooLua.h` / `TES3FooLua.cpp`
- CS editor objects: same shape with `CS` prefix instead of `TES3`
- NI scene graph types can exist in `SharedSE/` for shared primitives or in `MWSE/` when they are MWSE-specific/bound to Lua
- Lua events usually use `LuaFooEvent.h` / `LuaFooEvent.cpp`, with docs in `docs/source/events/foo.md` and autocomplete in `autocomplete/definitions/events/standard/foo.lua`
- Lua type docs/autocomplete usually follow the Lua-facing name: `docs/source/types/tes3foo.md`, `autocomplete/definitions/namedTypes/tes3foo.lua`
- `Debug/` and `Release/` subdirectories inside project folders are build outputs, not source landmarks

## Behavior Refinement

### Exposing Functions from Morrowind.exe

When implementing a function from IDA, do not use freefloating calls from other classes. Instead, implement wrapper functions on the associated class.

Bad result:

```cpp
using UpdateWaterReflectionFn = void(__thiscall*)(WaterController*);
const auto updateWaterReflection = reinterpret_cast<UpdateWaterReflectionFn>(0x51E150);
updateWaterReflection(waterController);
```

Good result:

```cpp
const auto TES3_WaterController_clearWaterReflectionFlag = reinterpret_cast<void(__thiscall*)(WaterController*)>(0x51E150);
void WaterController::clearWaterReflectionFlag() {
    TES3_WaterController_clearWaterReflectionFlag(this);
}
```

```cpp
waterController->updateWaterReflection();
```

### Expose Helper Functions as Member Functions

When adding helper functions, place them as member functions instead of namespaced static functions.

Bad result:

```cpp
namespace {
    void touchThunderReflection(const WeatherThunder* weather) {
        const auto dataHandler = controller ? controller->dataHandler : nullptr;
        const auto waterController = dataHandler ? dataHandler->waterController : nullptr;
        if (!waterController) {
            return;
        }

        dataHandler->waterController->clearWaterReflectionFlag();
    }
}
```

Good result:

```cpp
void Weather::clearWaterReflectionFlag() {
    const auto dataHandler = controller ? controller->dataHandler : nullptr;
    const auto waterController = dataHandler ? dataHandler->waterController : nullptr;
    if (!waterController) {
        return;
    }

    waterController->clearWaterReflectionFlag();
}
```
